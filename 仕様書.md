## 1. システム概要

### 1.1 目的
格闘ゲーム用ゲームパッドの入力を可視化し、コマンド入力の精度と反応速度を計測・表示するシステム。

### 1.2 主要機能
- リアルタイム入力表示（方向キー + ボタン）
- 入力履歴表示（最大12行）
- コマンド判定（波動拳、昇竜拳、スーパーアーツ）
- ボタン押下時間計測
- カメラプレビュー（QQVGA 160x120）
- 反応速度テスト（Action/React時間計測）
- コンボキャプチャ機能

### 1.3 システム特徴
- マルチコア並列処理による高速応答
- 1フレーム（16.67ms）単位の精度
- 低遅延設計（入力からMP通信まで）
- MP通信最適化（履歴表示は1F単位送信）

## 4. コア別機能仕様

- カメラストリーミング制御（30fps）

| バッファ数 | 1 |
| 共有メモリサイズ | 38,400 bytes |

#### 4.1.3 画像差分検出
| パラメータ | 値 | 説明 |
|-----------|-----|------|
| サンプリング間隔 | 5ピクセル | 処理負荷軽減 |
| サンプル数 | 768 (32x24) | 検出精度 |
| ピクセル差分閾値 | 30 | 色差分判定 |
| 変化ピクセル数閾値 | 15 | 動き判定 |

#### 4.1.4 ループ性能
- 目標周波数: 100kHz以上
- 実測周波数: 約124-126kHz

#### 4.1.5 USB電源管理
```
起動時: PIN_D02 = HIGH
    ↓
全コア初期化完了
    ↓

```

#### 4.1.6 LED制御
| LED | 機能 | ON条件 | OFF条件 |
|-----|------|--------|---------|
| LED0 | コントローラ接続状態 | 接続時 | 未接続時 |

### 4.2 SubCore1

#### 4.2.1 主要機能
- ゲームパッドUART受信
- 入力データ解析
- MainCoreへ転送

#### 4.2.2 入力データフォーマット（UART）
```
bit 23-16: reserved
bit 15-8:  btn (12bit ボタン状態)
bit 7-0:   dir (4bit 方向フラグ)
```

#### 4.2.3 方向フラグ定義
```
bit 0: UP    (0x1)
bit 1: DOWN  (0x2)
bit 2: LEFT  (0x4)
bit 3: RIGHT (0x8)
```

#### 4.2.4 接続状態通知
- 接続: `MSG_PAD_STATUS` (0x02)
- 切断: `MSG_PAD_STATUS` (0xFF)

## 5. 通信仕様

### 5.1 メッセージID一覧

| ID | 名称 | 方向 | 頻度 | 用途 |
|----|------|------|------|------|
| 1 | MSG_GAMEPAD | SC1→Main→SC2/SC3 | 入力変化時 | 入力データ |
| 2 | MSG_DRAW | Main→SC2 | 30fps | 現在入力表示 |
| 3 | MSG_SET_BUFFER | Main→SC2 | 起動時1回 | 共有メモリアドレス |
| 4 | MSG_CAMERA_READY | Main→SC2 | 30fps | カメラフレーム通知 |
| 5 | MSG_PAD_STATUS | SC1→Main→SC2 | 接続変化時 | 接続状態 |
| 7 | MSG_HISTORY | SC3→Main→SC2 | コンボ判定時 | 履歴色更新 |
| 8 | MSG_BUTTON_STATS | SC3→Main→SC2 | 1秒ごと | ボタン統計 |
| 9 | MSG_FPS_SC1 | SC1→Main→SC2 | 5秒ごと | SC1 FPS |
| 10 | MSG_FPS_SC3 | SC3→Main→SC2 | 1秒ごと | SC3 FPS |
| 11 | MSG_REACTION_START | Main→SC2 | テスト開始時 | 反応テスト開始 |
| 12 | MSG_REACTION_RESULT | Main→SC2 | 結果時 | 反応テスト結果 |
| 14 | MSG_REACTION_MARKER_DRAWN | SC2→Main | 描画完了時 | マーカー描画完了 |
| 15 | MSG_FPS_MAIN | Main→SC2 | 5秒ごと | MainCore Hz |

### 5.2 メッセージフォーマット

#### 5.2.1 MSG_GAMEPAD (ID:1)
```
bit 23-16: reserved
bit 15-8:  btn (16bit)
bit 7-0:   dir (8bit)
```

**送信頻度の最適化:**
- SubCore3へ: 入力変化時に即座に送信（ms精度維持）
- SubCore2へ: 約16.67ms（1F）ごとにまとめて送信（通信削減）

#### 5.2.2 MSG_DRAW (ID:2)
```
bit 31-22: frame (10bit, 0.1F単位)
bit 21-10: btn (12bit)
bit 9-6:   dir (4bit)
bit 5-0:   reserved
```

#### 5.2.3 MSG_HISTORY (ID:7)

**新規追加 (bit31=0):**
```
bit 31:    0 (新規フラグ)
bit 30-21: frame (10bit, 0.1F単位)
bit 20-17: dir (4bit)
bit 16-5:  btn (12bit)
bit 4-1:   color (4bit)
bit 0:     reserved
```

**色更新 (bit31=1):**
```
bit 31:    1 (更新フラグ)
bit 30-27: startIdx (4bit, 0-15)
bit 26-23: endIdx (4bit, 0-15)
bit 22-19: color (4bit)
bit 18-12: comboFrame (7bit, 0-127F)
bit 11-0:  reserved
```

#### 5.2.4 MSG_REACTION_RESULT (ID:12)
```
bit 31-16: 時間 (ms)
bit 15-2:  reserved
bit 1-0:   種別 (0=timeout, 1=Action, 2=React)
```

#### 5.2.5 MSG_BUTTON_STATS (ID:8)
```
bit 31-16: avgDuration (0.1ms単位)
bit 15-8:  sampleCount
bit 7-0:   reserved
```

### 5.3 通信最適化

#### 5.3.1 履歴表示用通信削減
**従来:**
- 入力変化ごとにSubCore2へ送信
- 1フレーム内に複数回送信の可能性

**改善後:**
- 1フレーム（約16.67ms）ごとに1回送信
- フレーム内の最終入力のみ送信
- 通信回数を大幅削減

#### 5.3.2 コンボ解析用通信
- SubCore3へは即座に送信（ms精度維持）
- 入力変化ごとに送信
- コンボ判定の精度を保証

## 7. コマンド判定仕様

### 7.1 波動拳（Hadouken）

#### 7.1.1 入力パターン
```
↓ → ↘ → + ボタン
```

#### 7.1.2 判定条件
| 条件 | 値 |
|------|-----|
| 方向キー完了猶予 | 11F |
| ボタン猶予 | 9F |
| 技フレーム対象 | ↘→のみ（↓は含まない） |

#### 7.1.3 許容パターン
- ↓↘→ + ボタン（基本）
- ↓↘→→ + ボタン（連続→OK）
- ↓↘ + ボタン（最後の→省略不可）

### 7.2 昇竜拳（Shoryuken）

#### 7.2.1 入力パターン
```
→ ↓ ↘ + ボタン
```

#### 7.2.2 判定条件
| 条件 | 値 |
|------|-----|
| 方向キー完了猶予 | 11F |
| ボタン猶予 | 8F |
| 技フレーム対象 | →↓↘（全て含む） |

#### 7.2.3 許容パターン
- →↓↘ + ボタン（正式）
- →↘↓↘ + ボタン（ショートカット）
- →N↓↘ + ボタン（途中にニュートラル）

### 7.3 スーパーアーツ（SA）

#### 7.3.1 入力パターン
```
(↓↘→ + ボタン) × 2
```

#### 7.3.2 判定条件
| 条件 | 値 |
|------|-----|
| 各波動拳の方向キー猶予 | 11F |
| 波動間ギャップ猶予 | 12F |
| ボタン猶予（最終） | 10F |

#### 7.3.3 技フレーム計算
```
SA技F = 1回目(↘→) + ギャップ + 2回目↓ + 2回目(↘→)
※1回目の↓は含まない
※ボタン猶予は含まない
```

### 7.4 クールダウン

#### 7.4.1 全コンボ共通クールダウン
- 時間: 800ms
- 対象: すべてのコンボ判定
- 目的: メッセージキュー飽和防止

#### 7.4.2 SA専用追加クールダウン
- 時間: 500ms
- 対象: スーパーアーツのみ
- 目的: 高負荷コンボの制限

### 7.5 フレーム計算

#### 7.5.1 内部計算
- 単位: 0.1F（小数点1桁）
- 1F = 16.67ms
- 計算式: `(elapsed_ms * 10 / 17) + 10`

#### 7.5.2 整数変換
- 関数: `toIntFrame()`
- 計算式: `(frameDecims + 5) / 10` （四捨五入）

## 8. 反応速度テスト仕様

### 8.1 概要
画面にマーカーを表示し、ユーザーの反応速度を計測する機能。

### 8.2 計測項目

#### 8.2.1 Action時間
**定義:** マーカー表示から、ユーザーの身体が動き始めるまでの時間

**計測方法:**
1. マーカー描画完了通知受信（SubCore2→MainCore）
2. 次のカメラフレームを基準画像として保存
3. 以降のフレームで差分検出
4. 閾値を超えたら時刻を記録

**カメラ差分検出:**
| パラメータ | 値 |
|-----------|-----|
| ピクセル差分閾値 | 30 |
| 変化ピクセル数閾値 | 15 |
| サンプリング | 5ピクセルごと（768点） |

#### 8.2.2 React時間
**定義:** マーカー表示から、ボタンを押すまでの時間

**計測方法:**
1. マーカー描画完了通知受信
2. ボタン入力受信（SubCore1→MainCore）
3. `millis()`で経過時間を計算

### 8.3 テスト制御

#### 8.3.1 タイミング
| パラメータ | 値 |
|-----------|-----|
| テスト間隔 | 5〜10秒（ランダム） |
| マーカー表示時間 | 2秒 |
| タイムアウト | 2秒（自動終了） |

#### 8.3.2 マーカー色
ランダムに選択（6色）:
- 赤 (0xF800)
- 緑 (0x07E0)
- 青 (0x001F)
- 黄 (0xFFE0)
- 紫 (0xF81F)
- 水 (0x07FF)

### 8.4 表示フォーマット
```
Action:   12.3 ms
React:   234.5 ms
```

**フォーマット仕様:**
- 整数部: 4桁、先頭スペース埋め
- 小数部: 1桁
- 単位: ms

### 8.5 状態遷移
```
待機 → テスト開始
      ↓
   マーカー表示要求（MainCore→SubCore2）
      ↓
   マーカー描画（SubCore2）
      ↓
   描画完了通知（SubCore2→MainCore）
      ↓
   計測開始
      ├→ Action検出（カメラ差分）
      └→ React検出（ボタン入力）
      ↓
   結果表示（SubCore2）
      ↓
   2秒後マーカー消去
      ↓
   待機（5〜10秒後に次のテスト）
```

### 8.6 遅延要因

#### 8.6.1 Action時間の遅延
- カメラフレームレート: 30fps (最大33ms遅延)
- 差分検出処理: 約0.1ms以下
- 合計: 設計上最大約33ms

#### 8.6.2 React時間の遅延
- UART通信: 数ms
- MP通信: 0.1ms以下
- 合計: 設計上数ms程度

## 付録A: トラブルシューティング

### A.1 カメラが映らない
**症状:** `Cam: 0 fps`、`SC2: Cam=0/s`

**原因:**
1. カメラモジュール接続不良
2. ケーブル断線
3. モジュール故障

**対処:**
1. カメラケーブルを再接続
2. Spresenseを再起動
3. シリアルログで初期化結果を確認

### A.2 システムが定期的にリブートする
**症状:** `_assert: Assertion failed panic`

**原因:**
1. メッセージキュー飽和
2. 過度な通信頻度
3. メモリ不足

**対処:**
1. クールダウン時間を増やす
2. ログ出力を削減
3. 不要な処理を削除

### A.3 履歴が表示されない
**症状:** 入力履歴が空白

**原因:**
1. MainCore→SubCore2の通信途絶
2. SubCore2のメッセージ処理不足

**対処:**
1. シリアルログで`MSG_GAMEPAD`送信を確認
2. SubCore2のメッセージ受信数を確認
3. 通信頻度を調整

### A.4 コンボが検出されない
**症状:** 正しいコマンド入力でも色が変わらない

**原因:**
1. クールダウン中
2. 入力タイミングが猶予外
3. SOCD処理により入力が無効化

**対処:**
1. 800ms以上待ってから再試行
2. より速く入力する
3. 同時押しを避ける
